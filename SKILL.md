# Podcast Analysis Doc

当用户提到以下意图时，使用这个 skill：

- “帮我分析这个播客”
- “把这期播客整理成导读/摘要/文档”
- “按‘关键词 + 全文摘要 + 章节速览’这种格式输出”
- “给你一个播客链接，帮我生成导读 docx”
- “把音频/转录稿/文稿整理成公众号导读或内部分析文档”
- “像《商业就是这样-春秋航空》那样拆一篇播客分析文档”

这个 skill 的目标不是随便做个摘要，而是把一段播客内容整理成 **结构稳定、可复用、可发布、可二次编辑** 的分析文档；如果用户要求 Word 成品，则继续交给 docx 能力导出 `.docx`。

默认交付物是这样的结构：

1. 标题
2. 节目信息 / 日期（如果有）
3. 关键词
4. 全文摘要
5. 章节速览（按时间戳分段）
6. 核心观点 / 商业启发（可选增强）
7. 可疑点 / 待核实点（如需要）
8. 导读 `.docx`（用户要求时）

## 使用前先判断输入类型

先识别用户给你的素材属于哪一类：

### 1. 纯文本 / Markdown / TXT / 已整理转录稿
直接读取并分析。

优先方式：
- 用 `read_file` 读取小文件
- 大文件只读必要片段，再做结构化总结

### 2. Word 文档（.docx）
这是本 skill 的常见输入。

处理原则：
- 先用专门的 docx 处理方式提取文本
- 如果环境里没有 `pandoc`，在 macOS 下可优先尝试 `textutil`
- 提取后再进入分析，不要直接凭文件名猜内容

### 3. PDF
交给 PDF 相关能力先提取文字，再回来分析。

### 4. 音频 / 视频
如果当前环境没有可靠转录能力，先问用户要：
- 转录稿
- 字幕文件
- 平台自动字幕导出结果

如果环境里已有稳定转录链路，再先转录，后分析。

### 5. 网页 / 节目链接
这是本 skill 重点支持的入口。

默认顺序：
- 先抓节目页标题、简介、show notes
- 再判断页面里是否有完整 transcript / 自动字幕 / 章节信息
- 如果只有简介，没有正文，不要硬写完整导读
- 正文不足时，向用户索要转录稿，或说明只能先做轻量版摘要

## 标准工作流

### 第一步：拿到可分析文本

无论输入是什么格式，先把它转成可读文本。

要求：
- 不直接基于零散截图或文件名输出结论
- 不跳过文本提取阶段
- 如果文本明显残缺，要明确告诉用户“当前只能做部分分析”

### 第二步：识别节目信息

先抽出这些基础信息：

- 节目标题
- 期数 / Vol 编号
- 日期
- 嘉宾 / 主播（如有）
- 主题
- 是否带时间戳
- 来源链接（如果用户给的是链接）

如果缺失，不要编造。

### 第三步：做主题归纳

先不要急着写正文，先确定：

- 这期播客核心在谈什么
- 主线论点是什么
- 支撑论点的关键分论点有哪些
- 节目结构是按时间推进、问题推进，还是案例推进

### 第四步：切章节

如果原文带时间戳，优先沿用。

如果原文没有时间戳：
- 可以按自然主题切段
- 但不要伪造具体分钟数
- 没有时间戳时，就写成“主题分段”而不是“章节速览时间轴”

### 第五步：生成分析文档

默认输出遵循模板：
- `templates/podcast_analysis_template.md`

如果用户要更偏“商业分析”，可额外增强：
- `核心观点`
- `商业启发`
- `值得继续追问的问题`
- `可验证数据点`

### 第六步：必要时导出为 docx

如果用户要 Word 文档：
- 先生成 Markdown / 纯文本结构稿
- 再读取 `docs/docx_export.md`
- 再调用 docx 能力生成 `.docx`
- 不要一上来在脑内拼格式

## 默认输出协议

### A. 标准版（最接近导读稿）

```markdown
# <节目名>_导读

<日期或来源信息>

## 关键词
<5~15 个关键词>

## 全文摘要
<1~2 段，概括节目核心结论、论证路径和价值>

## 章节速览
- <时间戳> <章节标题>
  <1 段总结>
- <时间戳> <章节标题>
  <1 段总结>
```

### B. 增强版（适合商业分析 / 内部知识库）

在标准版基础上增加：

```markdown
## 核心观点
- 
- 
- 

## 商业启发
- 
- 

## 待核实点
- 
```

### C. docx 交付版

当用户明确要 Word：
- 先完成标准版或增强版 Markdown 稿
- 再用 docx 能力导出成 `.docx`
- 文件名默认：`节目名_导读.docx`

## 质量要求

### 必须做到

- 基于已读到的文本输出，不要脑补节目内容
- 区分“播客里说了什么”和“你的分析判断”
- 时间戳必须来自原文或可靠来源
- 对不确定信息标记“待核实”
- 关键词要覆盖主题，不要只摘高频词
- 全文摘要要写出“结论 + 论证路径”，不是只堆名词
- 章节标题要可读，不要全写成平铺直叙
- 如果用户给的是链接，要明确当前导读基于的是 transcript、show notes 还是页面简介

### 禁止事项

- 禁止编造数据、嘉宾、时间戳、节目出处
- 禁止把广告口播、寒暄、无关闲聊误当成主论点
- 禁止只做“流水账复述”却声称是分析文档
- 禁止在证据不足时下过强结论
- 禁止在只有链接简介、没有正文时伪装成完整章节导读

## 推荐分析框架

适合商业 / 行业类播客时，优先检查这些维度：

1. 这期内容在回答什么问题
2. 论点是什么
3. 论据是什么
4. 数据或事实支撑是否充分
5. 商业模式 / 竞争格局 / 成本结构 / 用户需求是否被解释清楚
6. 主播或嘉宾有没有明显立场偏向
7. 哪些观点值得保留，哪些观点要谨慎引用

---

## 何时需要补问

出现这些情况时，不要硬输出完整成品，先补问：

- 只有音频，没有可用转录稿
- 文本只是一小段摘录，不足以代表全期内容
- 用户要“像示例文档一样”，但没说明要标准版还是增强版
- 用户要最终导出 `.docx`，但没有说明是否保留固定样式
- 用户只给了播客链接，但页面没有 transcript 或足够详细的 show notes

补问尽量最小化，例如：

- “你要我先出 Markdown 结构稿，还是直接出 Word 文档？”
- “现在手头只有部分文本，要不要我先做部分导读版？”
- “这个链接页没有完整 transcript，要不要你补一份转录稿，我再给你做完整 docx？”
