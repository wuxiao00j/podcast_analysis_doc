# Source Handling

这份文档专门讲 `podcast_analysis_doc` 如何处理不同输入来源。

## 0. 来源优先级

当用户同时给了多种材料时，正文分析优先级固定如下：

1. 用户提供的转录稿 / 字幕 / 音频转写文本
2. 音频文件转录得到的正文
3. 页面内完整 transcript
4. show notes / 章节提纲
5. 页面简介 / meta description / 标题卡片

规则：
- 页面信息主要用于补标题、日期、嘉宾、来源链接
- 页面信息也可用于判断缺口，例如“还缺完整 transcript”
- 页面信息**不能替代**音频或转录正文
- 如果用户给了音频或转录稿，不得跳过这些主材料，直接按页面简介写导读

## 1. 文本类输入

适用：
- `.txt`
- `.md`
- `.json` 中的 transcript 字段
- 已复制到对话里的转录稿

处理原则：
- 优先直接读文本
- 大文件先读关键区段
- 若用户只想要摘要，不必把全文全贴出来

## 2. Word 文档（.docx）

这是最常见的播客导读输入之一。

处理顺序：

1. 先提取纯文本
2. 检查原始结构有没有保留标题、时间戳、分段
3. 再进行分析

建议：
- macOS 下优先尝试 `textutil`
- 有 `pandoc` 时也可用 `pandoc`
- 如果提取文本明显缺段，要提醒用户

## 3. PDF

处理顺序：

1. 提取文本
2. 检查 OCR 质量
3. 如果乱码严重，告知用户结果不可靠

## 4. 音频 / 视频

优先级：

1. 已有转录稿 → 直接分析
2. 有字幕文件 → 先转成文本
3. 只有音频 / 视频 → 视环境能力决定是否先转录

如果当前环境没有稳定转录链路，不要假装能直接分析音频内容。

要向用户明确说明：
- 当前需要转录稿 / 字幕 / 平台自动字幕
- 拿到后可以继续整理成导读文档

**如果用户同时给了“音频 + 网页链接”，必须以音频转录内容作为正文主依据，网页链接只作为辅助元数据来源。**

## 5. 网页链接

如果用户给的是节目页链接：

- 先抓标题、简介、show notes
- 如果 show notes 足够详细，可以先做轻量摘要
- 如果用户要完整章节导读，通常仍需要转录稿
- 如果用户同时还给了音频或转录稿，网页内容不得喧宾夺主

## 6. 混合材料

常见情况：
- 节目简介 + 用户自己笔记
- 片段摘录 + 时间戳
- docx 摘要稿 + 外部网页链接
- 音频文件 + 节目页链接

处理原则：
- 主体分析基于最长、最完整、最接近正文的文本
- 其他材料只做补充，不喧宾夺主
- 如果不同材料互相冲突，要标注来源差异
- 如果既有音频转录正文，又有页面简介，默认以正文为准

## 7. 何时应停止并补问

遇到这些情况，应先补问：

- 文本明显只有一小部分
- 时间戳缺失，但用户又要求按分钟切段
- 节目标题、期数、来源都不清楚
- 用户要最终 docx，但没说明是否要固定版式
- 用户给了音频，但当前环境不能稳定转录

最小补问示例：

- “你要我先出 Markdown 分析稿，还是直接做 Word？”
- “现在这份只是摘要稿，不是完整转录。要不要我先按摘要稿做轻版导读？”
- “如果你要章节速览带时间戳，我需要原始转录里保留时间信息。”
- “你已经给了音频，但当前环境还没有稳定转录链路；要不要你补一份 transcript，或让我先只整理页面元信息？”
