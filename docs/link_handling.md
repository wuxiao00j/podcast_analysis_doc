# Link Handling

这份文档定义：当用户只给一个播客链接时，`podcast_analysis_doc` 应该怎么处理。

## 1. 默认目标

用户给链接，不等于已经给了足够文本。

你的第一目标不是立刻写导读，而是判断这个链接能不能提供：

- 节目标题
- 节目日期
- show notes
- 时间戳
- transcript / 自动字幕 / 正文

**补充原则：如果用户同时还给了音频、字幕或转录稿，链接页只负责补充元数据和查缺口，不能替代正文主依据。**

## 2. 推荐动作顺序

1. 打开链接页面
2. 抓标题、日期、节目简介
3. 查找 transcript / 字幕 / 展开全文入口
4. 判断信息量属于哪一级

### A. 完整 transcript 级
可以写完整导读，必要时导出 `.docx`。

### B. show notes 级
可以写轻量导读或摘要，但不要假装拥有完整正文。

### C. 简介页级
只能写一句话概览和缺口说明，通常应向用户补 transcript。

## 3. 推荐脚本入口

如果本地环境允许，优先用：

```bash
python3 scripts/fetch_podcast_page.py --url '<podcast-url>' --output report.json
```

这个脚本会输出：
- 页面标题
- meta description
- 发布时间（如果抓得到）
- 内容级别判断：`transcript_like / show_notes_like / summary_like`
- 时间戳样本
- 可疑 transcript / shownotes 链接线索
- 页面正文摘录

用途不是代替分析，而是帮助 agent **先判断链接质量是否够写完整导读**。

## 4. 信息不足时怎么说

建议直接说明：

- “这个链接页目前只有节目简介，没有完整 transcript。我可以先给你做轻量版导读，或者你补一份转录稿，我再给你做完整 docx。”
- “页面里有章节标题但没有正文，因此我能做结构提炼，但不能保证每一段都足够细。”

可直接参考模板：
- `templates/transcript_missing_response.md`

## 5. 什么时候可以直接继续

满足任一情况时，可以继续写完整导读：

- 页面里有完整 transcript
- 页面里有足够详细的分段文字和时间戳
- 用户额外补了转录稿 / 字幕 / 文本稿

## 6. 风险控制

绝对不要：

- 只靠节目简介写出像完整转录分析那样的章节导读
- 编造时间戳细节
- 把平台推荐文案当作节目原文

## 7. 如果页面需要登录

如果页面要登录或要展开隐藏 transcript：

- 优先提醒用户
- 必要时请用户先登录，再继续抓取
- 不要在未拿到正文时假装已完成分析
