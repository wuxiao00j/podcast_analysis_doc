# Workflow

这份文档把 `podcast_analysis_doc` 的执行流程写成可直接照着做的步骤。

## 0. 总原则

- 先拿到可分析文本，再开始写结论
- 先辨认节目结构，再写章节速览
- 先分清“节目内容”和“你的分析”，再写增强观点
- 信息不足时，不要硬凑成完整文档
- 时间戳只能来自原始文本或可靠来源
- 用户要 `.docx` 时，先产出 Markdown，再导出 Word
- **如果同时有音频 / 转录稿 / 页面信息，正文分析优先级必须是：音频转录 / 转录稿 > 页面 transcript > show notes > 页面简介**

## 1. 输入识别

先判断用户给的是哪一种：

1. 已整理文本
2. `.docx`
3. PDF
4. 音频 / 视频
5. 网页链接
6. 混合材料（节目简介 + 片段摘录 + 用户备注）

目标只有一个：拿到连续、可读、足够完整的文本。

## 2. 文本准备

### 已整理文本
- 直接读取
- 如果特别长，先扫开头、目录、时间戳和若干关键段

### docx
- 先提取纯文本
- 再判断结构是否保留了时间戳和标题层级

### PDF
- 先提取文字
- 再检查是否有 OCR 噪音

### 音频 / 视频
- 如果当前没有稳定转录手段，先请用户提供转录稿
- 如果已有转录稿，回到文本流程
- 如果用户同时给了链接，链接只用于补标题、日期、嘉宾、来源，不替代正文分析

### 网页链接
- 先抓标题、节目简介、show notes、时间戳、transcript 入口
- 判断正文是否足够
- 如果只有简介，没有 transcript，标记为“轻量信息输入”
- 如可执行，优先跑 `scripts/fetch_podcast_page.py`
- 如果页面是 JS 渲染、需要展开 transcript、需要滚动加载或需要登录，切到 `docs/browser_capture.md`
- **只有当页面里拿到了完整 transcript，或用户没有提供更高优先级正文时，页面正文才可作为主分析依据**

## 3. 基础信息抽取

至少抽这些字段：

- 节目标题
- 节目期数
- 日期
- 节目主题
- 主播 / 嘉宾（如有）
- 是否带时间戳
- 来源链接（如有）

原则：
- 有就写
- 没有就留空或省略
- 不编造

## 4. 结构识别

先回答这几个问题：

1. 这期节目主要在讨论什么
2. 核心结论是什么
3. 内容是怎么一步步展开的
4. 哪些段落是真正重要的
5. 哪些内容只是铺垫、寒暄或重复

### 有时间戳时
优先按时间戳切段。

### 没时间戳但有完整 transcript 时
按主题切段，不写伪时间。

### 只有简介 / show notes 时
只能输出轻量版摘要，不要伪装成完整章节导读。

## 5. 生成标准版文档

固定顺序：

1. 标题
2. 日期 / 来源
3. 关键词
4. 全文摘要
5. 章节速览

### 关键词
- 建议 5~15 个
- 要覆盖公司 / 行业 / 模式 / 问题 / 方法
- 不要只摘高频词

### 全文摘要
摘要至少要写出：
- 这期播客在讲什么
- 结论是什么
- 为什么成立
- 有什么价值

### 章节速览
每个章节建议包含：
- 时间戳或主题名
- 一个可读标题
- 一段 2~5 句的内容概述

## 6. 生成增强版文档

只有在以下情况才建议增加增强段落：

- 用户要行业分析
- 用户要做投资研究
- 用户要沉淀知识库
- 播客本身信息密度较高

增强项可包括：

- 核心观点
- 商业启发
- 值得追问的问题
- 待核实点
- 可验证数据点

## 7. 导出 docx

如果用户要 Word 成品：

1. 先完成 Markdown 结构稿
2. 再读 `docs/docx_export.md`
3. 优先运行 `scripts/export_docx.py`
4. 默认文件名：`节目名_导读.docx`
5. 如果无法可靠导出，就至少先交 Markdown，并明确差在哪一步

## 8. 输出前检查

至少检查这几件事：

1. 有没有编造时间戳
2. 有没有把模糊判断写成确定事实
3. 章节划分是否清晰
4. 摘要是不是只在复述，没有结论
5. 关键词是否覆盖主题
6. 如果有增强观点，是否明确属于分析判断
7. docx 是否和 Markdown 结构保持一致
8. 如果用户提供了音频或转录稿，正文是否真的基于这些主材料，而不是偷偷改成基于页面简介
