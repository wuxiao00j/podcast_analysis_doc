# Source Handling

这份文档专门讲 `podcast_analysis_doc` 如何处理不同输入来源。

## 1. 文本类输入

适用：
- `.txt`
- `.md`
- `.json` 中的 transcript 字段
- 已复制到对话里的转录稿

处理原则：
- 优先直接读文本
- 大文件先读关键区段
- 若用户只想要摘要，不必把全文全贴出来

## 2. Word 文档（.docx）

这是最常见的播客导读输入之一。

处理顺序：

1. 先提取纯文本
2. 检查原始结构有没有保留标题、时间戳、分段
3. 再进行分析

建议：
- macOS 下优先尝试 `textutil`
- 有 `pandoc` 时也可用 `pandoc`
- 如果提取文本明显缺段，要提醒用户

## 3. PDF

处理顺序：

1. 提取文本
2. 检查 OCR 质量
3. 如果乱码严重，告知用户结果不可靠

## 4. 音频 / 视频

优先级：

1. 已有转录稿 → 直接分析
2. 有字幕文件 → 先转成文本
3. 只有音频 / 视频 → 视环境能力决定是否先转录

如果当前环境没有稳定转录链路，不要假装能直接分析音频内容。

要向用户明确说明：
- 当前需要转录稿 / 字幕 / 平台自动字幕
- 拿到后可以继续整理成导读文档

## 5. 网页链接

如果用户给的是节目页链接：

- 先抓标题、简介、show notes
- 如果 show notes 足够详细，可以先做轻量摘要
- 如果用户要完整章节导读，通常仍需要转录稿

## 6. 混合材料

常见情况：
- 节目简介 + 用户自己笔记
- 片段摘录 + 时间戳
- docx 摘要稿 + 外部网页链接

处理原则：
- 主体分析基于最长、最完整的文本
- 其他材料只做补充，不喧宾夺主
- 如果不同材料互相冲突，要标注来源差异

## 7. 何时应停止并补问

遇到这些情况，应先补问：

- 文本明显只有一小部分
- 时间戳缺失，但用户又要求按分钟切段
- 节目标题、期数、来源都不清楚
- 用户要最终 docx，但没说明是否要固定版式

最小补问示例：

- “你要我先出 Markdown 分析稿，还是直接做 Word？”
- “现在这份只是摘要稿，不是完整转录。要不要我先按摘要稿做轻版导读？”
- “如果你要章节速览带时间戳，我需要原始转录里保留时间信息。”
